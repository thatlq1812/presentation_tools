## Overview

This document defines quality standards and constraints for AI-generated presentation content. All generated content must comply with these guidelines to ensure academic quality and professional presentation standards.

---

## 1. Tone and Voice

### 1.1. Academic Tone Requirements

| Attribute | Requirement |
| --- | --- |
| Voice | Third person preferred, first person plural (we) acceptable |
| Formality | Formal academic language |
| Objectivity | Evidence-based statements, avoid personal opinions |
| Precision | Specific terminology over vague descriptions |

### 1.2. Prohibited Elements

- Colloquial language or slang
- Emotional appeals without evidence
- Absolute claims without qualification ("always", "never", "best")
- Marketing or promotional language
- Icons, emojis, or decorative symbols
- Exclamation marks (except in direct quotes)

### 1.3. Recommended Language Patterns

| Instead of | Use |
| --- | --- |
| "This is the best method" | "This method demonstrates superior performance in X metric" |
| "Everyone knows that" | "Research indicates that" |
| "It's obvious that" | "Evidence suggests that" |
| "Amazing results" | "Significant improvement (X%)" |
| "We think" | "The analysis indicates" |

## 2. Content Depth and Analysis

### 2.1. Depth Levels

| Level | Description | Word Count Per Point | Use Case |
| --- | --- | --- | --- |
| High | In-depth analysis with evidence | 15-25 words | Research presentations, thesis defense |
| Medium | Balanced explanation | 10-15 words | Standard academic presentations |
| Low | Key points only | 5-10 words | Overview, summary slides |

### 2.2. Analysis Requirements

All content must demonstrate:

1. **Understanding:** Clear explanation of concepts
2. **Synthesis:** Connection between related ideas
3. **Evaluation:** Critical assessment when appropriate
4. **Application:** Practical relevance or implications

### 2.3. Evidence Standards

| Type | Requirement |
| --- | --- |
| Statistics | Include source and year |
| Quotes | Include author and publication |
| Claims | Support with referenced evidence |
| Comparisons | Use consistent metrics |

## 3. Slide Content Limits

### 3.1. Text Constraints

| Element | Minimum | Maximum | Optimal |
| --- | --- | --- | --- |
| Slide Title | 2 words | 8 words | 4-6 words |
| Bullet Point | 5 words | 15 words | 8-12 words |
| Bullets Per Slide | 2 | 6 | 3-5 |
| Total Words Per Slide | 20 | 60 | 30-50 |
| Speaker Notes | 50 words | 200 words | 100-150 words |

### 3.2. Content Balance Rules

1. **One Concept Per Slide:** Each slide should focus on a single main idea
2. **Rule of Three:** Group related points in threes when possible
3. **Progressive Disclosure:** Build complexity gradually across slides
4. **Visual Balance:** Text should occupy no more than 60% of slide area

### 3.3. Readability Standards

| Metric | Target |
| --- | --- |
| Average Sentence Length | 15-20 words |
| Passive Voice Usage | Less than 20% |
| Jargon Density | Define all technical terms on first use |
| Acronym Usage | Spell out on first use, then abbreviate |

## 4. Structural Requirements

### 4.1. Presentation Flow

A well-structured presentation must include:

| Section | Percentage of Total | Purpose |
| --- | --- | --- |
| Introduction | 10-15% | Context, objectives, roadmap |
| Background/Literature | 15-20% | Foundation knowledge |
| Main Content | 50-60% | Core arguments and analysis |
| Results/Discussion | 10-15% | Findings and implications |
| Conclusion | 5-10% | Summary and future work |

### 4.2. Slide Sequencing Rules

1. **Opening Hook:** First content slide should capture attention
2. **Logical Progression:** Each slide should lead naturally to the next
3. **Signposting:** Use section headers to guide audience
4. **Recap Points:** Summarize after major sections
5. **Strong Close:** Final slides should reinforce key message

### 4.3. Transition Guidelines

Between slides, content should:

- Maintain logical flow
- Build upon previous points
- Avoid redundancy
- Prepare for upcoming content

## 5. Visual Content Guidelines

### 5.1. Image Requirements

| Requirement | Specification |
| --- | --- |
| Relevance | Directly supports content |
| Quality | High resolution, professional |
| Source | Include attribution |
| Alt Text | Required for accessibility |
| Caption | Descriptive, includes source |

### 5.2. Diagram Standards

| Type | When to Use |
| --- | --- |
| Flowchart | Processes, workflows, decision trees |
| Architecture | System structure, components |
| Timeline | Chronological events, evolution |
| Comparison | Side-by-side analysis |
| Hierarchy | Organization, taxonomy |

### 5.3. Table Guidelines

- Maximum 5 columns, 6 rows
- Clear headers
- Consistent formatting
- Highlight key data points
- Include unit of measurement

## 6. Code Content Guidelines

### 6.1. Code Block Requirements

| Requirement | Specification |
| --- | --- |
| Length | Maximum 10 lines per slide |
| Language Tag | Always specify language |
| Comments | Include for complex logic |
| Formatting | Consistent indentation |
| Highlighting | Mark key lines |

### 6.2. Code Explanation Rules

1. Every code block must have accompanying explanation
2. Explain what the code does, not just how
3. Highlight key concepts or patterns
4. Reference line numbers when explaining

## 7. Citation and References

### 7.1. In-Slide Citations

| Style | Format |
| --- | --- |
| APA | (Author, Year) |
| IEEE | [Number] |
| Harvard | (Author Year) |

### 7.2. Reference Slide Format

```
## References

1. Author, A. A. (Year). Title of work. Publisher/Journal.
2. Author, B. B., & Author, C. C. (Year). Title. Journal Name, Volume(Issue), Pages.
```

### 7.3. Citation Requirements

- Minimum 3 references for academic presentations
- Recent sources preferred (within 5 years unless foundational)
- Mix of types: journals, books, conference papers
- Verify all URLs are accessible

## 8. Speaker Notes Guidelines

### 8.1. Purpose

Speaker notes serve as:

1. Extended explanation for presenter
2. Context not shown on slide
3. Timing cues
4. Transition hints

### 8.2. Content Requirements

| Element | Description |
| --- | --- |
| Elaboration | Expand on bullet points |
| Examples | Specific cases or scenarios |
| Questions | Anticipated audience questions |
| Timing | Suggested time per point |
| Transitions | How to move to next slide |

### 8.3. Format

```markdown
<!-- Speaker Notes:
Main Point: Explain the core concept in 2-3 sentences.

Details:
- First sub-point with additional context
- Second sub-point with example

Timing: Spend approximately 1.5 minutes on this slide.

Transition: This leads into the next section on implementation.

Potential Question: Audience may ask about scalability.
Answer: Our approach scales linearly with input size.
-->
```

## 9. Quality Checklist

Before finalizing generated content, verify:

### Content Quality
- [ ] All claims are supported by evidence
- [ ] Technical terms are defined
- [ ] No grammatical or spelling errors
- [ ] Consistent terminology throughout

### Structure Quality
- [ ] Logical flow between slides
- [ ] Appropriate layout for content type
- [ ] Balanced visual and text content
- [ ] Clear section transitions

### Academic Standards
- [ ] Proper citations included
- [ ] Formal tone maintained
- [ ] No prohibited elements (emojis, slang)
- [ ] Objective language used

### Technical Compliance
- [ ] Layout tags correctly formatted
- [ ] All required fields populated
- [ ] Speaker notes included for complex slides
- [ ] Metadata file complete

## 10. Common Mistakes to Avoid

| Mistake | Description | Solution |
| --- | --- | --- |
| Wall of Text | Too much text on one slide | Split into multiple slides |
| Orphan Bullets | Single bullet under heading | Add more points or remove heading |
| Inconsistent Depth | Varying detail levels | Maintain uniform depth |
| Missing Context | Jumping to conclusions | Build foundation first |
| Jargon Overload | Too many undefined terms | Define terms on first use |
| Weak Conclusions | Vague or missing takeaways | State clear implications |
| Missing Transitions | Abrupt topic changes | Add bridging content |
| Over-decoration | Unnecessary visual elements | Keep design minimal |
